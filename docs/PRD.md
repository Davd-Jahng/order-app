# COZY 커피 주문 앱 — PRD (Product Requirements Document)

---

## Part I. 개요

### 1.1 문서 정보

| 항목 | 내용 |
|------|------|
| 프로젝트명 | COZY — 커피 주문 앱 |
| 문서 목적 | 주문하기·관리자 화면 기능 및 데이터 요구사항 정의 |
| 범위 | 풀스택 웹앱 (주문 플로우 + 관리자 운영) |

### 1.2 프로젝트 목적

- **사용자**: 커피 메뉴 선택, 옵션 선택, 장바구니 담기, 주문 제출
- **관리자**: 주문 상태 관리, 재고 조회·조정, 대시보드 요약 확인

### 1.3 개발 범위

| 구분 | 내용 |
|------|------|
| **포함** | 주문하기 화면, 관리자 화면, 메뉴/옵션/주문/재고 CRUD, 공통 네비게이션 |
| **제외** | 사용자 인증, 결제 기능 |
| **메뉴** | 커피 메뉴만 대상 (추후 확장 가능) |

### 1.4 가이드라인 및 제약사항

- 프론트엔드와 백엔드는 **별도 디렉터리/프로젝트**로 개발
- 기본 웹 기술 위주로 구현

## Part III. 기능 요구사항

### 3. 주문하기 화면

#### 3.1 화면 구성

| 영역 | 설명 |
|------|------|
| 상단 헤더 | [2.1] 공통 헤더 (주문하기 탭 강조) |
| 메뉴 영역 | 상품 카드 가로 배치 |
| 장바구니 영역 | 담긴 항목 목록, 총 금액, 주문하기 버튼 |

#### 3.2 메뉴(상품) 카드

- **구성 요소**: 상품 이미지(또는 플레이스홀더), 상품명, 기본 가격, 간단한 설명, 옵션(체크박스), 담기 버튼
- **옵션 예시**: 샷 추가 (+500원), 시럽 추가 (+0원) — 선택 시 단가에 반영
- **담기**: 클릭 시 선택 옵션을 반영한 항목이 장바구니에 추가

**비즈니스 규칙**

- 동일 상품 + 동일 옵션 조합 → 장바구니에서 **수량 합산**
- 단가 = 기본가 + 옵션 추가금

#### 3.3 장바구니

- **목록**: `상품명 (옵션명) X 수량` + 항목별 금액
- **총 금액**: 합계 표시
- **주문하기 버튼**: 클릭 시 장바구니 내용으로 주문 생성 (결제·인증 없음)

### 4. 관리자 화면

#### 4.1 화면 구성

| 영역 | 설명 |
|------|------|
| 상단 헤더 | [2.1] 공통 헤더 (관리자 탭 강조) |
| 관리자 대시보드 | 주문 상태별 건수 요약 |
| 재고 현황 | 상품별 재고 수량 및 증감 조절 |
| 주문 현황 | 개별 주문 목록 및 상태 변경 |

#### 4.2 관리자 대시보드

- **표시 형식**: `총 주문 N / 주문 접수 N / 제조 중 N / 제조 완료 N`
- **의미**: 주문 상태별 건수 집계 (총 주문, 접수, 제조 중, 제조 완료)

#### 4.3 재고 현황

- **카드**: 상품명, 현재 재고 수량(예: 10개), `-` / 수량 / `+` 버튼
- **동작**: `+` 1 증가, `-` 1 감소 (재고 0 미만 불가)
- **저장**: 즉시 반영 또는 저장 버튼 — 구현 시 결정

#### 4.4 주문 현황

- **행 정보**: 주문 시각, 주문 내용(상품 x 수량), 금액, 상태 버튼(예: 주문 접수)
- **상태 흐름**: 신규 → 주문 접수 → 제조 중 → 제조 완료
- **정렬**: 최신순 등 (구현 시 결정)

**비기능**

- 재고 수량 ≥ 0 유지

## Part IV. 데이터 요구사항

### 5.1 엔티티·속성 요약

| 엔티티 | 주요 속성 | 비고 |
|--------|-----------|------|
| **메뉴(상품)** | 이름, 기본 가격, 설명, 이미지 URL(또는 경로) | |
| **옵션** | 옵션명, 추가 금액 (0원 가능) | 상품과 연관 |
| **장바구니** | 상품 ID, 옵션(또는 옵션 조합), 수량, 계산된 단가 | 클라이언트/세션 또는 DB |
| **주문** | 주문 시각, 주문 항목 스냅샷(상품/옵션/수량/금액), 총 금액, **상태** | 상태: 접수대기/접수/제조중/제조완료 |
| **재고** | 상품 ID, 현재 수량 | 상품별 1행 |

### 5.2 주문 상태

- 접수 대기 → 주문 접수 → 제조 중 → 제조 완료  
- 대시보드는 위 상태별 건수 조회·집계

### 5.3 데이터 스키마 (백엔드·DB)

#### menus (메뉴)

| 컬럼명 | 타입 | 필수 | 설명 |
|--------|------|:----:|------|
| id | PK (정수/UUID) | ○ | 메뉴 고유 식별자 |
| name | VARCHAR | ○ | 커피명 |
| description | TEXT |  | 설명 |
| price | INTEGER | ○ | 가격 (원) |
| image | VARCHAR |  | 이미지 URL 또는 경로 |
| stock | INTEGER | ○ | 재고수량 (기본 0 이상) |

#### options (옵션)

| 컬럼명 | 타입 | 필수 | 설명 |
|--------|------|:----:|------|
| id | PK (정수/UUID) | ○ | 옵션 고유 식별자 |
| name | VARCHAR | ○ | 옵션명 |
| extra_price | INTEGER | ○ | 옵션가격 (원, 0 가능) |
| menu_id | FK → menus.id | ○ | 연결할 메뉴 |

#### orders (주문)

| 컬럼명 | 타입 | 필수 | 설명 |
|--------|------|:----:|------|
| id | PK (정수/UUID) | ○ | 주문 고유 식별자 |
| created_at | TIMESTAMP | ○ | 주문일시 |
| menu_id | FK → menus.id | ○ | 커피(메뉴) |
| menu_name | VARCHAR |  | 커피명 (스냅샷, 선택) |
| option_id | FK → options.id |  | 옵션 |
| option_name | VARCHAR |  | 옵션명 (스냅샷, 선택) |
| quantity | INTEGER | ○ | 수량 |
| amount | INTEGER | ○ | 금액 (원) |
| status | VARCHAR | ○ | 주문처리상태: 주문접수 / 제조중 / 제조완료 |

*주문 1건에 항목이 여러 개일 경우: 아래 order_items 사용 또는 orders에 1주문 1행씩 여러 행 저장.*

#### order_items (주문 상세, 선택)

| 컬럼명 | 타입 | 필수 | 설명 |
|--------|------|:----:|------|
| id | PK (정수/UUID) | ○ | 주문 항목 식별자 |
| order_id | FK → orders.id | ○ | 주문 |
| menu_id | FK → menus.id | ○ | 메뉴 |
| menu_name | VARCHAR |  | 커피명 스냅샷 |
| option_id | FK → options.id |  | 옵션 |
| option_name | VARCHAR |  | 옵션명 스냅샷 |
| quantity | INTEGER | ○ | 수량 |
| amount | INTEGER | ○ | 금액 (원) |

## Part VI. 백엔드 개발 사양

### 7.1 데이터 모델 요약

| 테이블 | 용도 |
|--------|------|
| **menus** | 메뉴(커피) 마스터. 재고수량은 관리자 재고 현황과 연동 |
| **options** | 옵션 마스터. 메뉴와 N:1 또는 N:M 관계로 연결 |
| **orders** | 주문 저장. 1건 = 1행 또는 헤더+상세 구조 선택 |

*상세 스키마(컬럼명·타입·필수·설명)는 [5.3 데이터 스키마 (백엔드·DB)](#53-데이터-스키마-백엔드db) 참고.*

### 7.2 데이터 흐름

1. **메뉴 표시**  
   - **menus** 데이터를 조회하여 **주문하기 화면**과 **관리자 화면**(재고 현황 등)에 표시한다.
2. **장바구니**  
   - 사용자가 주문하기 화면에서 **커피명·옵션**을 선택해 담으면, 선택 정보는 **장바구니**에 표시된다. (장바구니는 프론트엔드 상태 또는 세션)
3. **주문 저장**  
   - **주문하기** 버튼 클릭 시 주문 정보를 **orders**에 저장한다.
4. **주문 현황**  
   - **orders** 정보를 관리자 화면의 **주문 현황** 섹션에 표시한다.
5. **주문 상태**  
   - 주문의 **기본 상태**는 **주문접수**이다.  
   - 관리자가 클릭하면 **제조중** → **제조완료** 순으로 변경한다.

### 7.3 API 설계

| 구분 | 설명 | API 요약 |
|------|------|----------|
| **메뉴 조회** | 주문하기(메뉴) 클릭/진입 시 DB **menus**를 불러와 화면에 표시 | `GET /api/menus` — 메뉴 목록 반환 (필요 시 options 포함 또는 별도 조회) |
| **주문 저장** | 주문하기 버튼 클릭 시 **orders**에 저장 | `POST /api/orders` — 주문 payload 전송 후 DB orders에 저장 |
| **재고 차감** | 주문 처리 상태가 **완료**로 바뀌면 해당 주문 수량만큼 재고에서 차감 | `PATCH /api/orders/:id` (상태를 제조완료로 변경) 시 서버에서 **menus** 재고수량 차감 처리 |

- **GET /api/menus**  
  - 응답: 메뉴 목록(커피명, 설명, 가격, 이미지, 재고수량 등). 옵션은 별도 `GET /api/options` 또는 메뉴별 옵션 포함 구조로 제공.
- **POST /api/orders**  
  - 요청: 주문일시, 커피명(또는 menu_id), 옵션, 수량, 금액, (초기) 주문처리상태(주문접수 등).  
  - 응답: 생성된 주문 ID 등.
- **주문 상태 변경 및 재고 차감**  
  - `PATCH /api/orders/:id`로 상태를 **제조완료**로 변경할 때, 서버에서 해당 주문의 메뉴·수량에 따라 **menus**의 재고수량을 차감한다.  
  - (선택) 재고 조회·수정용 `GET /api/menus`(재고 포함), `PATCH /api/menus/:id`(재고 수동 조정) 등.

## Part VII. 부록

### 8. 용어 정의 (프론트·백 공통)

| 용어 | 설명 |
|------|------|
| 담기 | 메뉴+옵션을 장바구니에 추가하는 행위 |
| 주문하기(버튼) | 장바구니 내용을 한 건의 주문으로 제출하는 행위 |
| 옵션 | 샷 추가·시럽 추가 등 상품별 추가 선택 및 추가 금액 |
| 주문 접수 | 관리자가 주문을 확인하고 접수 상태로 변경하는 행위 |
| 제조 중 | 접수 후 음료 제조가 진행 중인 상태 |
| 제조 완료 | 주문 음료 제조가 완료된 상태 |
| 재고 현황 | 메뉴(상품)별 보유 수량 및 증감 조정 기능 |
